# YACReader iOS 迁移计划

更新时间：2026-03-20

## 1. 目标与范围

本项目的目标，是在当前 iOS 工程内，用 `SwiftUI + UIKit` 重建并整合桌面版 `YACReader` 与 `YACReaderLibrary` 的核心能力，最终形成一个单体 iOS App：

- 不实现 `YACReaderLibraryServer`
- 不再维持桌面版那种“资料库应用 + 独立阅读器 + 本地 IPC”的拆分模式
- iOS 端直接把资料库浏览、漫画管理、漫画阅读整合到同一个 App 内
- 尽可能兼容桌面版现有资料库格式、封面缓存、阅读进度、元数据结构
- 核心功能必须先完整落地：
  - 漫画阅读
  - 漫画资料库管理
  - 扫描入库
  - 本地导入（单本 / 目录）
  - `SMB` 在线浏览与直接阅读
  - `SMB` 导入到本地资料库（单本 / 目录）
  - 阅读进度与书签
  - 搜索、收藏、标签、阅读列表

非目标：

- `YACReaderLibraryServer`
- 托盘、菜单栏、桌面快捷键编辑器等桌面专属能力
- 平台特有的外部命令调用方式，例如桌面版的第三方阅读器命令

这份计划会作为后续执行的主线文档。随着移植推进，允许调整阶段边界、实现顺序和技术细节，但不改变“先把核心链路做通，再逐步补齐功能”的总体策略。

## 4. iOS 平台约束与策略

### 4.1 文件系统与权限

iOS 最大的差异，不是 UI，而是文件访问模型：

- 用户不能像桌面一样给我们一个永久可写的任意路径
- 外部文件夹访问通常依赖 `UIDocumentPicker` 与 `security-scoped bookmarks`
- 文件提供器、iCloud Drive、外接存储、第三方云盘的读写能力和稳定性不一致
- 后台长时间扫描、实时文件监听都比桌面受限

因此必须设计“双存储策略”：

1. `In-Place Library`
   - 如果用户选中的资料库目录可写，就继续使用桌面兼容结构：
   - `<library>/.yacreaderlibrary/library.ydb`
   - `<library>/.yacreaderlibrary/covers/*`

2. `Mirrored Metadata Library`
   - 如果源目录不可写，App 在 `Application Support` 建立镜像元数据目录
   - 原始漫画文件仍从用户授权目录读取
   - 数据库、封面、索引、缓存写在 App 私有目录
   - 通过 `security-scoped bookmark` 记录源目录

这两个模式都要实现，前者保证兼容桌面生态，后者保证 iOS 环境下可用性。

### 4.3 依赖策略

迁移阶段不再默认“所有东西自己重写”。iOS 端的依赖选择按这条优先级推进：

1. 优先用系统框架
   - 例如 `PDFKit`、`ImageIO`、`SQLite3`、`UIDocumentPicker`、`QuickLookThumbnailing` 可直接承担的能力，不额外引库
2. 系统能力不足时，优先引入成熟第三方
   - 只要能显著减少维护成本、并且适合 iOS 打包与离线运行，就优先使用第三方而不是继续手写实现
3. 只有第三方仍不满足时，才保留自实现
   - 主要用于资料库兼容语义、桌面历史行为对齐，或必须保留的 archive / schema 细节

当前依赖决策：

- 已接入：`SWXMLHash 8.1.1`
  - 方式：本地 vendoring
  - 用途：解析压缩包中的 `ComicInfo.xml`
- 已接入：`SMBClient 0.3.1`
  - 方式：本地 vendoring
  - 用途：`SMB` 登录、连接 share、列远程目录、下载单本漫画到本地缓存
- 暂不切换：`GRDB.swift`
  - 原因：当前 SQLite 兼容层已经覆盖大量桌面 schema 细节，整体替换会引入一次大规模数据库层重构；等扫描/增量更新链路稳定后再评估迁移
- 暂不切换：`ZIPFoundation`
  - 原因：当前 `ZIP/CBZ` 阅读链路已可用，近期收益不如补扫描和阅读器核心体验；后续若要补 `ZIP64`、更完整加密 ZIP 兼容，可再考虑切换
- 待评估：`PINCache`
  - 方向：封面磁盘缓存、阅读页二级缓存
  - 时机：优先在真机性能验证后决定是否引入

### 4.2 后台与性能

iOS 上必须把以下问题当成一等公民：

- 大型资料库首次扫描耗时长
- 大分辨率页面解码占内存
- 压缩包随机读取比桌面更容易触发卡顿
- App 进入后台后任务可能被中断

应对策略：

- 所有扫描、封面提取、页码统计都放到后台任务队列
- 页面图像采用“两级缓存”：内存页缓存 + 磁盘缩略图缓存
- 阅读器只保留有限页面缓冲，避免整本解码进内存
- 长任务统一可取消、可恢复、可分片
- 后台刷新只做轻任务，重扫描以手动触发为主

### 4.3 产品形态差异

桌面版的某些能力要换表达方式：

- 树状边栏：iPad 使用 sidebar / split view，iPhone 使用 drill-down 导航
- 右键菜单：改为 `contextMenu`、编辑模式、长按菜单
- 拖拽重排：iPad 优先支持，iPhone 用编辑模式补齐
- 快捷键：后续补硬件键盘命令，不先做“快捷键编辑器”
- 放大镜、页流、信息面板等用 UIKit 容器承接，SwiftUI 做外层编排

### 4.4 顶层信息架构（修订版）

随着本地资料库、远程 `SMB` 浏览、在线阅读与导入路径持续增多，当前“全部挂在首页 / 资料库首页”已经开始过载。后续顶层导航调整为 3 个底部 tab：

- `书库`
  - 面向“已经进入本地资料库体系”的内容
  - 负责：
    - 多资料库入口
    - 继续阅读
    - 收藏 / 阅读中 / 最近
    - 标签 / 阅读列表
    - 本地导入（单本 / 目录）
  - 目标：让“管理已经拥有的漫画”成为最短路径
- `浏览`
  - 面向“发现与搬运内容”的工作流
  - 负责：
    - `SMB` 服务器列表
    - 远程目录浏览
    - 远程漫画缩略图浏览
    - 远程在线阅读
    - 从 `SMB` 导入到本地资料库（单本 / 目录）
  - 目标：把“浏览远程源”和“导入本地库”解耦，不再塞进书库首页
- `设置`
  - 面向全局偏好与维护
  - 负责：
    - 阅读偏好
    - 导入策略
    - 缓存与远程下载管理
    - 调试 / 关于 / 数据维护入口
  - 目标：把低频系统级操作从核心业务页面抽离

这套信息架构的设计约束：

- 首页必须保持简洁，避免一个页面同时承担“资料库管理 + 远程浏览 + 导入 + 设置”
- `SMB` 在线浏览不是附属功能，而是核心能力，与本地书库并列
- 视觉上坚持现代原生 iOS / iPadOS 风格：
  - 以 `TabView`、`NavigationStack / NavigationSplitView`、系统材质与层级清晰的卡片为主
  - 避免继续叠加桌面式密集工具栏
  - 列表 / 网格 / 详情关系要清楚，优先保证主链路可读性

### 5.4 关键技术选型

#### UI

- 外层导航与状态管理：`SwiftUI`

- 大图阅读与手势缩放：`UIScrollView` + 自定义 `UIViewController`

  - iPad：`NavigationSplitView` / `UISplitViewController`
  - iPhone：`NavigationStack`

#### 并发

- `Swift Concurrency`

#### 数据库

- 初版直接使用 `SQLite3`
- 精确复刻桌面 schema 与 migration 逻辑
- 若后续需要更高开发效率，可在不破坏兼容层的前提下引入轻量封装

#### 底层桥接

- `Objective-C++`
- 封装 `compressed_archive` 的能力为 iOS 可调用接口
- PDF 直接优先使用 `PDFKit`

## 6. 数据兼容计划

### 6.1 必须保持兼容的内容

- `.yacreaderlibrary/library.ydb`
- `.yacreaderlibrary/covers/*.jpg`
- `.yacreaderlibrary/id`
- `comic_info` 字段语义
- `folder/comic` 路径语义
- 漫画哈希规则
- 封面生成规则
- `Favorites / Reading / Recent / label / reading_list` 结构

### 6.2 明确保留的桌面规则

#### 哈希

- 继续使用桌面版 `pseudoHash`
- 规则：读取文件前 `512KB` 做 `SHA1`，再拼接文件大小
- 这样 iOS、桌面、已有资料库可互认同一本漫画

#### 封面

- 继续沿用桌面版缩放逻辑：
  - 横向图：宽度缩到 `640`
  - 纵向图：正常情况缩到宽度 `480`
  - 超长封面：高度缩到 `960`
  - JPEG 质量 `75`

#### 数据库

- 初版直接兼容桌面 `9.16.0` 结构
- iOS 专属 UI 状态不写回 `library.ydb`
- iOS 专属临时状态放到 App 私有辅助数据库或偏好存储

### 6.3 资料库存储模式

#### 模式 A：原地兼容模式

- 适用于用户给出的目录可写
- 直接创建或更新 `<root>/.yacreaderlibrary`
- 与桌面版互通最好

#### 模式 B：镜像元数据模式

- 适用于目录只读、提供器不允许隐藏目录写入、或写入不稳定
- 源目录只负责提供漫画文件
- `library.ydb` 与 `covers/` 放在 App 沙盒：
  - `Application Support/Libraries/<uuid>/`
- 路径仍保存相对路径，保持模型一致

### 6.4 兼容迁移策略

执行顺序：

1. 优先实现“读已有桌面库”
2. 再实现“创建新库”
3. 再实现“更新现有库”
4. 最后做“导出/导入包”和更高级的跨模式迁移

## 7. 功能优先级矩阵

### 7.1 iOS 版产品取舍

从当前实现阶段开始，iOS 版不再追求“桌面功能逐条照抄”，而是按下面原则排序：

- 高频核心链路必须优先：
  - 导入/建库
  - 浏览/搜索
  - 阅读
  - 进度/书签
  - 收藏/阅读中/最近/标签/阅读列表
- 中频管理功能放第二层：
  - 扫描更新
  - 元数据编辑
  - ComicInfo.xml
  - 自定义封面
- 低频且高实现成本的桌面遗留功能移出核心计划，降级到 backlog 或观察项：
  - 放大镜
  - 自动滚动 / Zigzag
  - 亮度 / 对比度 / Gamma 及桌面图像滤镜兼容
  - 资料库打包导入导出
  - Comic Vine 在线抓取
  - 层级阅读子列表与拖拽重排

这些能力不是永久删除，而是不再阻塞 iOS 版核心发布。

- 已继续补厚远程错误恢复链路：当 SMB 目录因连接、认证或 share 问题无法打开时，错误页除了 `Open Last Comic` 之外，还会在有多本离线副本时提供 `Offline Shelf` 入口，减少错误态下只能依赖单一本最近漫画的限制
- 已开始把远程缓存策略产品化：`Settings` 里的 `Remote Browse Cache` 现可直接切换 `Compact / Balanced / Extended` 三档缓存预设，远程下载副本的自动淘汰不再只是内部常量，后续可继续扩展到更细粒度的离线策略
- 已开始收敛远程浏览的同步磁盘开销：`RemoteReadingProgressStore` 现会缓存远程阅读会话，`RemoteServerBrowsingService` 现会缓存各服务器与全局的下载副本摘要；`Remote Servers` 与远程目录页也改成优先消费这些快照，减少列表渲染时重复扫描缓存目录和重复读取远程阅读进度文件

## 9. 关键实现策略

### 9.1 阅读器策略

阅读器是整个项目的技术核心，必须单独强调。

#### 页面管线

- 打开文档时只建立页目录，不立即解码全书
- 当前页优先，前后页预加载
- 解码后的页面进入 LRU 内存缓存
- 超出预算的页面回收
- 磁盘缓存只缓存封面、缩略图，不缓存整本全尺寸页

#### 布局模式

- `Single Page`
- `Double Page`
- `Double Page RTL`
- `Continuous Vertical`，用于 webcomic
- `Fit Width`
- `Fit Height`
- `Fit Screen`
- `Full Resolution`

#### 手势

- 单击显示/隐藏工具栏
- 双击缩放
- 双指缩放
- 拖拽平移
- 左右/上下翻页
- 页缩略条拖动跳页

#### 进度与状态

- 翻页后节流写回进度
- App 后台、切换漫画、关闭阅读器时强制写回
- 同步更新：
  - `currentPage`
  - `hasBeenOpened`
  - `read`
  - `lastTimeOpened`
  - `bookmark1..3`
  - `imageFiltersJson`

### 9.2 搜索策略

搜索不只做一个输入框。

### 9.3 标签与阅读列表策略

桌面版标签和阅读列表不是装饰功能，而是资料库管理核心，必须保真。

实现要点：

- 保留桌面 schema
- 保留特殊列表语义
- 支持批量加入/移除
- 支持顺序重排
- 维持标签颜色语义
- 维持顶级列表和子列表层级

### 9.4 元数据策略

元数据系统分三层：

1. 本地扫描获得的基础信息
   - 页数
   - 封面
   - 哈希
   - 相对路径
   - 类型

2. 嵌入 XML
   - ComicInfo.xml
   - ComicRack / ComicTagger 风格数据

3. 手工编辑 / 在线抓取
   - Comic Vine
   - 用户修改封面
   - 用户修改评分、标签、简介等

策略上要保证：

- 手工编辑不会被扫描轻易覆盖
- XML 导入是受控的
- Comic Vine 回填可以预览后确认

## 11. 风险清单

| 风险 | 严重度 | 说明 | 应对 |
| --- | --- | --- | --- |
| RAR/7z 在 iOS 的底层支持不稳定 | 高 | 桌面可依赖外部/本地库，iOS 需要静态集成 | 尽早验证 bridge，必要时分格式逐个打通 |
| 外部目录写权限不稳定 | 高 | 某些文件提供器不允许隐藏目录写入 | 实现 mirrored 模式兜底 |
| 大图和大库导致内存压力过高 | 高 | 阅读器和网格都容易爆内存 | 严格分页缓存、缩略图缓存、性能测试前置 |
| 数据库兼容实现不严谨会污染现有桌面库 | 高 | 一旦写错 schema 影响用户历史数据 | 初版以“只读打开旧库”验证通过后再放开写入 |
| 后台扫描被系统打断 | 中 | iOS 不适合桌面式长任务 | 支持取消、恢复、分段更新 |
| QML 桌面体验直接照搬到 iPhone 会很差 | 中 | 交互密度和信息密度不同 | iPhone 和 iPad 分别设计呈现方式 |
| Comic Vine API 和速率限制 | 中 | 网络能力不稳定 | 放在后期实现，做错误隔离 |
